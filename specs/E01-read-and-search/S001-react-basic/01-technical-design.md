# E01-S001: 技术设计

> ReACT 基础版的设计说明，重点解释结构、职责和关键决策，不重复展开实现代码。

---

## 1. 整体结构

这一版的核心结构可以拆成四个部分：

1. **LLM Client**
   - 负责和模型通信
   - 当前选用 Anthropic SDK

2. **ReACT Loop**
   - 负责驱动“调用模型 -> 解析结果 -> 执行工具 -> 继续循环”
   - 是整个 Agent 的核心控制流

3. **Tools**
   - 当前只保留两个只读工具：`read_file`、`list_directory`
   - 它们构成最小的查看闭环

4. **CLI / Agent 入口**
   - 把核心能力暴露成一个可以运行和观察的交互入口

当前最值得先看的代码入口：

- `packages/core/src/loop.ts`
- `packages/core/src/agent.ts`
- `packages/core/src/tools/read-file.ts`
- `packages/core/src/tools/list-directory.ts`
- `packages/tui/src/cli.ts`

## 2. LLM 客户端设计

### 2.1 选型

当前使用 `@anthropic-ai/sdk`。

主要理由：

- 官方维护，类型定义完整
- 原生支持 Tool Use
- 支持 `baseURL` 配置，后续可以切换兼容提供商

### 2.2 当前配置边界

本 Story 只处理最小可用配置：

- `ANTHROPIC_API_KEY`
- `ANTHROPIC_BASE_URL`（可选）
- `MODEL_NAME`（可选）

这一版不追求复杂的多模型抽象，只先把一个可运行闭环打通。

### 2.3 消息组织方式

这里采用 Anthropic 的 content blocks 思路：

- 用户消息进入模型
- 模型返回文本或工具调用
- 如果触发工具调用，就执行工具
- 工具结果再回传给模型，继续下一轮

关键不是消息格式细节本身，而是要理解：

> 工具结果不是流程外的副产物，而是下一轮推理的输入。

## 3. 工具设计

### 3.1 为什么只做两个工具

本 Story 只实现：

- `read_file`
- `list_directory`

原因很简单：这两个工具放在一起，才构成最小“看项目”的能力包。

### 3.2 read_file 的设计重点

关注点主要有三个：

- 支持按路径读取文件
- 支持按行号范围读取
- 输出带行号，方便模型和学习者引用内容

### 3.3 list_directory 的设计重点

关注点主要有三个：

- 支持列出目录内容
- 支持递归模式
- 输出完整相对路径，方便后续直接继续读取文件

### 3.4 错误处理原则

当前工具出错时，不抛复杂异常给上层，而是尽量返回清晰的错误文本。

这样做的目的不是“工程最优”，而是：

- 让模型看到失败原因
- 让循环可以继续推进
- 让当前 MVP 的控制流保持简单

## 4. ReACT 循环设计

### 4.1 核心流程

这一版的循环只做最关键的几步：

1. 接收用户输入
2. 调用模型
3. 判断是否结束
4. 如果是工具调用，则执行工具
5. 把工具结果回传给模型
6. 继续下一轮，直到结束

### 4.2 为什么这是 Agent，而不只是一次模型调用

因为这里已经出现了一个真正的闭环：

- 模型不是一次性给完最终答案
- 它可以借助工具获取新信息
- 然后基于新信息继续判断下一步

这就是本 Story 最关键的认知升级。

### 4.3 当前终止条件

当前主要关心三类终止或分支信号：

| 条件 | 处理 |
|------|------|
| `end_turn` | 正常结束，返回最终文本 |
| `tool_use` | 执行工具并继续循环 |
| `max_tokens` | 提前返回当前内容 |

另外还保留了最大循环次数，用于防止无限循环。

## 5. 设计决策记录（ADR）

### ADR-01：使用 Anthropic SDK

**决策**：当前版本使用 `@anthropic-ai/sdk`。

**理由**：

1. 官方维护，类型较完整
2. Tool Use 机制清晰
3. 方便后续切换到兼容 API 提供商

### ADR-02：工具错误返回文本而不是复杂对象

**决策**：工具失败时优先返回可读错误文本。

**理由**：

1. 让模型能直接看到错误信息
2. 保持循环逻辑简单
3. 更符合当前 MVP 阶段的教学节奏

### ADR-03：目录结果展示完整路径

**决策**：目录输出保留完整路径信息。

**理由**：

1. 模型后续可以直接继续引用
2. 学习者更容易看懂“从目录到文件”的推进关系
3. 对后续搜索和读取能力更友好

---

## 6. 当前不做的事情

这一版明确暂不处理：

- 路径安全检查
- 文件写入等危险操作
- 更复杂的错误恢复与重试
- 更重的工具注册抽象
- 流式输出

这些会留给后续 Story 或后续 Epic。
